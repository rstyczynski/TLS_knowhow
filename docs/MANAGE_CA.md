## Context

Internet communication is secured with Transport Layer Security (TLS), which relies on a certificate chain of trust: a Root CA signs an Intermediate CA, and the Intermediate CA signs the service (leaf) certificate. Clients trust the leaf certificate only if they can build this chain to a trusted root in their local trust store. Because major OS and browser root programs include selected Root CAs, those certificates are distributed by default and provide baseline trust.

Systems typically use public Root CAs for internet-facing services. For internal services, organizations often use private CAs (self-managed trust) to reduce cost and simplify internal certificate issuance. In this model, organizations are responsible for registering internal Root CA certificates in all internally used computer systems.

Following this practice, each system interplays with two types of Root CAs: official and internal. The former is maintained by official OS releases and patches, and the latter is managed internally.

This document elaborates on the management of internal Root CAs.

> **Note** Management of Root and Intermediate CA certificates is a separate concern from managing server (leaf) certificates; this document focuses on the handling of roots, as their trust and provisioning differs from that of leaf certificates and intermediates which are typically bundled with leaf ones.

## Stakeholder scenarios

### GitHub Runner

Self-hosted GitHub Runners communicate with internet services through a proxy server. The proxy handles internal communication using its own root certificate. This certificate must be registered in the operating system so that all utilities and services used during GitHub workflow execution can trust it.

The current solution registers all known certificates in the OS image used to run the runners before the compute instance starts.

A request was made to design an extension that supports runtime certificate injection, instead of relying only on deployment-time certificate injection.

### OCI Function

During development of a Node.js-based OCI Function, the developer encountered a TLS issue caused by an internal CA used by the proxy path.

A request was made to design an extension that supports runtime certificate injection instead of disabling certificate verification.

## Initial analysis

### Main concern

A Root CA certificate is essentially a text file, typically in PEM format, containing cryptographic and identity information generated by standard tools such as OpenSSL. To establish trust, this certificate can be either installed into the operating system’s trusted certificate store or supplied directly to individual HTTPS-enabled applications.

Linux systems (such as Red Hat and CentOS) include system libraries and tools for CA management, most notably the `update-ca-trust` framework, which allows administrators to add or remove trusted root certificates and manage blocklists. These changes are reflected in the system trust store so applications use or reject certificates accordingly.

Many libraries and command-line tools (such as curl, Node.js) also allow you to specify additional trusted certificate files at runtime using environment variables or configuration settings. This enables flexible management of trust without requiring root access or permanent system changes.

### Required concerns

Beyond certificate registration, system owners should maintain regular certificate hygiene to address security leaks and certificate expiry. In the first case, if private keys used to sign internal certificates are leaked, affected certificates must be revoked immediately and permanently, because attackers can use them for man-in-the-middle attacks. In the second case, overlooked certificate expiry can disrupt systems, because TLS communication will fail. Both scenarios should be handled with clear operational procedures to ensure security and business continuity.

Certificates that are expired (or close to expiration) should trigger the platform owner to contact the issuer. Therefore, self-issued certificates should always include contact information in certificate subject or extension fields.

## Detailed analysis

This analysis addresses the problem of internally generated certificates, with the goal of supplying required certificates to runtime processes. Secondary use cases include blocking compromised certificates and tracking outdated ones.

### Operating system

Red Hat Linux provides convenient certificate management through a directory-based interface for the `update-ca-trust` framework. System administrators place certificates in the `anchors` and `blocklist` directories, then run the `update-ca-trust` utility.

This is considered a **pre-deploy** (image-build-time) procedure when performed during image build (before the compute instance starts). The same may be applied to container images by instrumenting the build via Dockerfile directives.

**Deploy** time assumes certificates are registered in the operating system during the first run of the compute instance. At this stage, an Ansible playbook can be used with elevated privileges (for example, the `opc` user in Oracle OCI environments).

**Runtime in-band** procedure is controlled by the system workflow, which executes a CA update step early in the process. Registration can use the same techniques as deploy time, with elevated privileges (for example, `sudo`) to run `update-ca-trust`. A dedicated Ansible playbook should standardize and automate this procedure with built-in privilege escalation. Existing long-running processes may still require reload or restart to pick up updated trust data.

**Runtime out-of-band** reconfiguration is performed by system administrators using standard Ansible operations over `ssh` to a group of target compute instances. This approach is typically used to respond to security incidents (for example, compromised certificates) and to perform housekeeping tasks such as removing expired certificates.

### Runtime environment

When configuring the operating system is not possible or not practical, operators can configure certificate trust at the process level via environment variables or using the specific API of the library in use.

**Environment variables** may be used to configure root certificates before the process starts. Process-level configuration typically overwrites the system-level CA store, which requires merging the provided certificates with the system ones. Node.js is a partial exception: `NODE_EXTRA_CA_CERTS` appends to its bundled CAs automatically, but when a `ca` option is passed explicitly in code (e.g. via undici's `connect.ca` or `requestTls.ca`), the built-in bundle is replaced and the caller must spread `tls.rootCertificates` alongside any custom CA.

The following table enumerates environment variables per runtime. OpenSSL settings work for all utilities that use this library, and the same applies to Go, which influences Terraform. When working at this level, one needs to investigate which internal library handles secure communication. Note that Windows Schannel ignores regular OpenSSL settings.

| Runtime | Variable | Behavior |
| --- | --- | --- |
| OpenSSL (CLI/apps linked with OpenSSL) | `SSL_CERT_FILE=file`, `SSL_CERT_DIR=dir` | OpenSSL-level CA bundle or CA directory override |
| curl | `CURL_CA_BUNDLE=file` | curl-specific CA bundle path override (ignored by Windows/Schannel backend) |
| curl | `SSL_CERT_FILE=file`, `SSL_CERT_DIR=dir` | CA file/directory override for curl (ignored by Windows/Schannel backend) |
| Terraform / Go | `SSL_CERT_FILE=file`, `SSL_CERT_DIR=dir` | Go x509 override on Unix; OpenSSL-style names, but not an OpenSSL backend |
| Node.js | `NODE_EXTRA_CA_CERTS=file` | appends to built-in CAs; read lazily at first TLS use |
| Python (`requests`) | `REQUESTS_CA_BUNDLE=file` | replaces default bundle for `requests` library |
| Java | `javax.net.ssl.trustStore` (JVM property) | replaces default cacerts; on RHEL often aligned with system trust |

> Node.js requires more explanation, as its implementation is unique. By default it is shipped with bundled root certificates (Mozilla NSS); it can be extended with system and custom certificates using `NODE_USE_SYSTEM_CA=1` and `NODE_EXTRA_CA_CERTS` environment variables [3](https://nodejs.org/en/learn/http/enterprise-network-configuration).

**Runtime libraries** can typically be configured to override system behavior. It is a broad subject, and this analysis is limited to Node.js in the context of HTTPS and proxy servers.

The current HTTP client library for Node.js is **undici** (npm package `undici`), the HTTP/1.1 and HTTP/2 client developed by the Node.js core team and bundled internally since Node.js v18. Undici powers the native `fetch()` available in Node.js v18+ and is the underlying engine for `@octokit/rest` v21+. The legacy `node:https` module (callback-based, no built-in proxy support) is considered superseded and is not covered here.

Undici initializes TLS with Node.js's bundled Mozilla NSS root certificates. The operating system trust store is not used by default, but may be enabled with the `--use-system-ca` flag or `NODE_USE_SYSTEM_CA=1` environment variable. When enabled, Node.js loads system CAs in addition to its bundled CAs for TLS validation [3](https://nodejs.org/en/learn/http/enterprise-network-configuration). When `NODE_USE_SYSTEM_CA=1` is combined with `NODE_EXTRA_CA_CERTS`, the full set of certificates — system, bundled, and custom — is available.

Custom CA certificates are supplied via undici agent options: `connect.ca` in an `Agent` (direct connections) and `requestTls.ca` in a `ProxyAgent` (tunneled connections). The agent is configured once and applied globally via `setGlobalDispatcher`, so all subsequent `fetch()` calls use it automatically. Custom certificates are typically supplied at runtime via Base64-encoded environment variables, with `tls.rootCertificates` spread alongside to preserve the full Mozilla NSS trust bundle.

To analyze any runtime's ability to handle custom certificates, follow this investigation sequence:

1. Determine whether the runtime uses OpenSSL or has its own internal TLS stack.
2. Identify environment variables that supply additional CA certificates (e.g., `SSL_CERT_FILE`).
3. Investigate whether the runtime's TLS library is configurable at the API level:
   a) reading the runtime's built-in root certificates,
   b) supplying custom certificates alongside them.

## Design

This document focuses on a subset of the problem: 

1. GitHub Runner, which is Oracle OCI boot volume image
2. Container configuration for Node.js

### GitHub Runner compute instance

The GitHub runner operates in one of multiple regions (3 realms, 2 regions each). A design question is whether all realms and regions use the same internal Root CA or follow another rule. A single Root CA is a reasonable assumption, but a different layout may exist and could complicate runner image generation. To support flexible Root CA installation in the GitHub Runner image, the following is needed:

1. A way to obtain the current certificate(s)
2. A standard, repeatable way to obtain the current certificate(s)
3. An update to the image generation workflow to add the certificate(s) dynamically

When image-level configuration is not practical, an alternative is to install certificates during the Terraform VM creation process. To support flexible Root CA installation on the runner VM, the following is needed:

1. A way to obtain the current certificate(s)
2. A standard, repeatable way to obtain the current certificate(s)
3. An Ansible role that installs and activates the certificate(s) on Linux
4. An Ansible playbook that runs this role, embedded in the runner VM creation Terraform workflow

Other options are to implement a GitHub Action or to rely on environment variables; the latter is more complex because it requires technology-specific variables (e.g. Go, Python). Environment-based configuration may be used for special cases, but the two approaches above (image-level and Terraform/Ansible) are generally more robust.

### OCI Function Docker for Node.js

Node.js running in Docker should be configured using an **image-build-time** approach. The base image should include the OS CA bundle (e.g. the `ca-certificates` package) so that Node can use the system trust store in addition to bundled NSS trust store. To enable the system trust store in Node.js, set the `NODE_USE_SYSTEM_CA=1` environment variable.

On top of that, internal root certificates must be provided as a PEM bundle file referenced by `NODE_EXTRA_CA_CERTS` (the variable accepts a file path, not inline PEM content). To support flexible Root CA installation in the Node.js Docker image, the following is needed:

1. A way to obtain the current certificate(s)
2. A standard, repeatable way to obtain the current certificate(s)
3. A `Dockerfile` step to install the OS CA bundle (`ca-certificates`)
4. A development-time procedure (CLI tool or Git hook) to update the internal certificates bundle file in the build context
5. A `Dockerfile` step to copy the internal certificate bundle into the image and configure `NODE_EXTRA_CA_CERTS` to point to it
6. A `Dockerfile` step to configure Node.js to use system CAs (`NODE_USE_SYSTEM_CA=1`) in addition to the bundled store

Alternatively, the build workflow may inject these steps into the image as part of a GitHub workflow (rather than committing updated certificate bundles to the repository). The workflow should use the same CLI tool that developers use locally.

CLI tool is independent i.e. detects already performed configuration of Dockerfile and does changes only when it's needed. Following this approach tool may be executed by both developer and the workflow.

The idea is to keep all Root Certificate configuration inside a single, delimited block in the Dockerfile so the CLI (or the GitHub workflow) can reliably detect it and update only that section. The block is marked with `# START::Root Certificate configuration section` and `# STOP::Root Certificate configuration section`.

Exemplary Dockerfile:

```dockerfile
FROM node:24-slim

# START::Root Certificate configuration section
COPY certs/custom_certs.pem /etc/ssl/certs/custom_certs.pem
ENV NODE_EXTRA_CA_CERTS=/etc/ssl/certs/custom_certs.pem

RUN apt-get update && apt-get install -y ca-certificates && rm -rf /var/lib/apt/lists/*
ENV NODE_USE_SYSTEM_CA=1
# STOP::Root Certificate configuration section

WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .

CMD ["node", "server.js"]
```

Build:

```bash
touch certs/custom_certs.pem
docker build -f Dockerfile.node -t ssl-node .
```

### Application level Node.js Fn configuration

When the container image is not prepared and proxy configuration is required, the developer may use a workaround by supplying the extra CA certificate via function configuration. The solution decodes the certificate from a Base64-encoded environment variable and injects it into undici's TLS options at runtime — no file on disk needed.

The recommended approach uses undici's `ProxyAgent` with `requestTls.ca`. Spreading `tls.rootCertificates` alongside the custom CA preserves the full Mozilla NSS bundle; omitting it would restrict trust to the custom CA alone.

```js
import { ProxyAgent, setGlobalDispatcher } from 'undici';
import tls from 'node:tls';

const proxyCaB64 = process.env.PROXY_CA_B64;
if (!proxyCaB64) {
  console.error('[error] PROXY_CA_B64 is not set');
  process.exit(1);
}
const proxyCa = Buffer.from(proxyCaB64, 'base64').toString('utf8');

const agent = new ProxyAgent({
  uri: process.env.HTTPS_PROXY ?? 'http://127.0.0.1:8888',
  requestTls: { ca: [...tls.rootCertificates, proxyCa] },
});
setGlobalDispatcher(agent);

// All fetch() calls now use the proxy with the custom CA trusted
const res = await fetch('https://api.example.com/endpoint');
```

`requestTls.ca` applies to the inner tunneled TLS connection (after the HTTP CONNECT tunnel is established). For direct connections without a proxy, use `connect.ca` in an undici `Agent` instead:

```js
import { Agent, setGlobalDispatcher } from 'undici';
import tls from 'node:tls';

const extraCa = Buffer.from(process.env.EXTRA_CA_B64, 'base64').toString('utf8');

const agent = new Agent({
  connect: { ca: [...tls.rootCertificates, extraCa] },
});
setGlobalDispatcher(agent);
```

> **undici `connect` vs `requestTls`**: `connect` applies to TLS with the target server (direct connections) or with the proxy itself. `requestTls` applies to the inner TLS connection after the HTTP CONNECT tunnel is established. For an HTTP proxy, only `requestTls` matters.

This approach works transparently for any library built on native `fetch()` — no extra package or wrapper is needed.

## Backlog

### Generic

- Design a standard, repeatable mechanism to obtain the current internal certificate(s).

### Runner: deploy-time certificate injection

- Create an Ansible role that installs and activates certificates on Linux.
- Create an Ansible playbook embedding the role in the Terraform runner VM creation workflow.

### Runner: build-time certificate injection

- Clarify whether all realms/regions share one internal Root CA or use separate ones.
- Update the runner image generation workflow to inject certificates dynamically.

### Node.js Docker: CLI tooling

- Build an idempotent CLI tool that adds or updates the Root Certificate configuration block (`START::` / `STOP::`) in a Dockerfile.
- Integrate the CLI tool into the GitHub workflow so certificate injection runs automatically before each build.

### Certificate hygiene

- Define standard requiring embedding a contact point (at least e-mail) to each issued certificate
- Define an expiry monitoring procedure with an escalation path to the certificate issuer.
- Define an operational procedure for immediate revocation when a private key is compromised.
